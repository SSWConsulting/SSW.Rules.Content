You finish a Claude Code session at 5pm on Tuesday. It knows what your coding preferences are, the test command, and the auth module quirk you corrected twice.

You start a new session Wednesday morning and it tries the wrong command again.

**Coding agents forgetting useful project context between sessions.** If you're a developer using Claude Code or Codex, the solution is a simple 3-step.

<endIntro />

**Memory** means persisted project context that a coding agent can load in a future session. It can be a markdown instruction file, an automatic note Claude Code saved, or a structured task graph like Beads.

## The Goldfish Memory

LLMs are stateless. Every model call starts fresh unless the tool around it supplies previous context. A coding agent only "remembers" because something reloads useful text or structured state into the context window.

<imageEmbed alt="Image" size="medium" showBorder={false} figurePrefix="none" figure="AI losing memory and starts hallucinating" src="/uploads/rules/v2-ai-agent-memory-systems/memory_comic.png" />

The everyday symptoms:

* Re-explaining the build command every session
* Re-correcting the same coding style mistake every session
* The agent rediscovering the same bug it already debugged last week
* Long features that need multiple sessions to ship — and the agent forgetting where it was halfway through

For example:

| Scenario                         | Stateless coding agent                                       | Coding agent with memory                                      |
| -------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------- |
| New session in an existing repo  | Tries `npm test`, even though the repo uses `pnpm test:unit` | Reads the test command from `CLAUDE.md` / `AGENTS.md`         |
| Reopening a bug from last week   | Repeats the same failed debugging path                       | Sees the saved note about the actual auth-token expiry issue  |
| Continuing a feature after lunch | Needs you to restate the plan and the files already changed  | Loads the current task, decisions, and blocked/unblocked work |

You don't need a vector database or a knowledge graph to fix this for normal client engineering. You need three things, in order.

## From Goldfish to Dolphin

### Step 1: Write repo instructions (⭐️ recommended baseline)

A plain markdown file at the repo root is the baseline. This is where you put **things every coding agent should know on day one** — build commands, test commands, coding conventions, architectural decisions, and the things you'd otherwise type into chat every time.

```text
your-project/
├── CLAUDE.md            # Claude Code reads this
├── AGENTS.md            # Codex/Factory/etc. read this
├── .claude/
│   └── rules/           # path-scoped rules — only load when matching files are touched
│       ├── api.md       #   e.g. paths: ["src/api/**/*.ts"]
│       └── testing.md
└── CLAUDE.local.md      # personal, gitignored
```

A few things that catch devs out:

* **Claude Code reads `CLAUDE.md`, NOT `AGENTS.md`.** If your client repo already uses `AGENTS.md` (because they use Codex / Factory / mixed tools), the canonical pattern is a one-line `CLAUDE.md` containing `@AGENTS.md`. Both tools then read the same source of truth.
* **Keep it under \~200 lines.** Longer files consume context and reduce adherence. If it's growing, move topic-specific rules into `.claude/rules/*.md` with `paths:` frontmatter so they only load when matching files are touched.
* **Be specific.** *"Use 2-space indentation"* and *"Run `npm test` before committing"* work. *"Format code nicely"* and *"Test your changes"* don't.
* **Run `/init` once** in any new client repo. Claude analyses the codebase and drafts a starter `CLAUDE.md` from it.

### Step 2: Turn on your tool's local memory (⭐️ recommended baseline)

Repo instructions are the shared source of truth. Tool-local memory is the private notebook your coding agent keeps as you work. It is **not** the same thing as editing `CLAUDE.md` / `AGENTS.md`, and it is **not** just the current chat transcript.

Use it for learnings the tool discovers during real work: the command that actually passes, the debugging insight from yesterday, the style nudge you've repeated twice, or the local gotcha that is useful to you but not worth committing to the repo.

For Claude Code, this is **auto memory**:

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # concise index — loaded into every session (first 200 lines / 25KB)
├── debugging.md       # detailed notes — loaded on demand when Claude needs them
└── api-conventions.md
```

For Codex, this is **Codex Memories**:

```text
~/.codex/memories/
├── MEMORY.md           # consolidated memory
├── memory_summary.md   # compact summary loaded for future sessions
├── raw_memories.md     # extracted memories from previous work
└── rollout_summaries/  # per-session summaries
```

|                  | Repo instructions (you write)                  | Tool-local memory (tool writes)                                             |
| ---------------- | ---------------------------------------------- | --------------------------------------------------------------------------- |
| Examples         | `CLAUDE.md`, `AGENTS.md`                       | Claude Code auto memory, Codex Memories                                     |
| What it contains | Instructions and rules                         | Learnings, preferences, gotchas, previous-session context                   |
| Lives in         | Your repo (shared with the team via git)       | Your machine (`~/.claude/` or `~/.codex/`)                                  |
| Loaded into      | Every session, depending on the tool           | Future sessions, depending on tool settings                                 |
| Best for         | Coding standards, build commands, architecture | Debugging insights, preferences the tool discovered, local workflow gotchas |

Setup checklist:

* **For Claude Code, auto memory is on by default.** Toggle via `/memory` in a session, or `"autoMemoryEnabled": false` in settings, or `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`.
* **For Codex, check whether Memories are enabled.** Run `codex features list` and look for `memories`. In config-based installs, memory settings live under `~/.codex/config.toml`.
* **Different words → different store.** Saying *"remember X"* writes to tool-local memory. Saying *"add this to CLAUDE.md"* or *"add this to AGENTS.md"* edits repo instructions instead.
* Audit tool-local memory occasionally. It is machine-local context, so it can drift or capture something you would rather move into repo instructions.

  </>}

### Step 3: Add Beads when work needs a task graph (⭐️ recommended for long projects)

Steps 1 and 2 help the agent remember **context**. Beads helps the agent remember **work**.

When a feature spans multiple sessions, markdown plans start to decay. The agent creates `phase-3-plan.md`, then a nested `phase-1-of-phase-3-plan.md`, then forgets which level of the plan it is in after compaction. You end up with lots of partially true plans and no reliable answer to *"what is ready to do next?"*

This is what [Beads](https://github.com/gastownhall/beads) is for. It is a distributed graph issue tracker for coding agents, powered by [Dolt](https://github.com/dolthub/dolt). It gives agents a dependency-aware issue graph, JSON-friendly commands, ready-work detection, claiming, audit history, and a small project memory mechanism via `bd remember`.

Use Beads for:

* **Long-horizon planning:** break a big feature into issues instead of nested markdown plans
* **Dependency tracking:** ask `bd ready` for unblocked work instead of rereading prose
* **Discovered work:** when the agent finds a bug mid-task, it files an issue instead of losing it
* **Session handoff:** a fresh agent can run `bd prime` and recover the current workflow context
* **Multi-agent coordination:** agents can claim different ready issues without duplicating work

```bash
# Install once (system-wide)
brew install beads

# In your project
bd init
bd setup codex               # Codex CLI - creates/updates AGENTS.md
bd setup claude              # Claude Code - installs hooks/settings

# Core agent workflow
bd prime                     # print workflow context and persistent memories
bd ready                     # list unblocked work
bd update bd-a1b2 --claim    # claim a task
bd remember "Decision: ..."  # store project memory for future bd prime output
```

You'll know it's time when:

* `CLAUDE.md` is past 200 lines and growing
* You're nudging *"remember this"* multiple times a session
* The agent keeps re-deriving the same architectural decision incorrectly
* You're losing work because the agent ignores "unrelated" problems it discovered mid-task
* The project has nested workstreams the agent gets lost in across compactions

## Honorable mentions

You probably have came across Mem0, Zep, Letta, MemoryGraph, Cognee, and LangMem. These are not bad tools. They are usually aimed at a different memory problem from "help my coding agent remember this repo":

* **Mem0** — useful for **assistant products with many end-users**, where each user needs their own extracted preferences, facts, and long-term state.
* **Zep / Graphiti** — useful for **temporal knowledge graphs**, especially when you need to ask *"what was true on date X?"* or reason over relationships that change over time.
* **Letta (formerly MemGPT)** — useful for **long-running autonomous agents** that manage their own memory over days or weeks, rather than a developer-controlled coding session.
* **MemoryGraph** — useful when you want a **graph-based MCP memory server** for explicit relationship tracking across entities, decisions, and notes.
* **LangMem / Cognee** — useful when you're already building on **LangChain / LangGraph** or ingesting large unstructured document sets that need retrieval and memory workflows.

<boxEmbed
  body={<>
    **Rule of thumb:** pick the memory system that matches the shape of the thing being remembered. Repo conventions, developer preferences, and agent work items fit the 3-step stack above. End-user state, temporal facts, autonomous agents, graph relationships, and document-scale retrieval may justify a dedicated memory product.
