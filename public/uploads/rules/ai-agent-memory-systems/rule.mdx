An LLM is stateless. Every call is a fresh start. Anything you want the model to "know" past the current turn has to be put back into the prompt somehow. That's it. "AI Agent memory" is just different answers to one question: **what do we store, where do we put it, and how do we get it back into the context window at the right moment?** The problem is, the "memory" word covers at least three different problems — project context, per-user personalisation, and long-running autonomy — and picking the wrong tool wastes weeks.

## What kinds of memory are there?

People conflate two separate questions when they talk about agent memory: **the cognitive types** (what kind of knowledge is this?) and **the architectural patterns** (how is it stored and retrieved?). They're different axes — every real system maps to a point on both.

The cognitive types come from the CoALA framework (Princeton, 2023):

* **Working / in-context** — the current conversation window. RAM. Close the session and it's gone.
* **Episodic** — what happened before. "The user asked about Resend in March."
* **Semantic** — generalised facts. "Gordon prefers self-hosted tooling."
* **Procedural** — how to do things. Workflows, learned routines, the agent's "muscle memory."

  body={<>
    Most production agents only really need two: **working memory** (handled by the LLM provider) plus some flavour of **long-term memory** that blends the other three. Treating them as separate stores is usually over-engineering until you have a concrete reason.
  </>}

The architectural patterns are the *how*:

* **Markdown files** - a `.md` file gets dumped into the system prompt at session start. Human-readable, git-versioned, deterministic.
* **Vector store** - embed every memory, pull top-k closest, scale well, fuzzy on contradictions and time.
* **Knowledge graph** - typed nodes and edges (`Gordon WORKS_AT SSW`). Some add validity windows for "what was true on date X" queries.
* **Agent-managed paging** - the agent itself decides what to page in/out using explicit tool calls. OS-style core + archival memory.
* **Hybrid extraction layers **- an LLM extracts memorable facts from raw conversation, deduplicates, and stores in a mix of vector + structured stores.

Almost every product on the market is a mix of these.

## How do you give your AI Agent memory?

There are dozens of options. Here are the ones that matter.

### CLAUDE.md / AGENTS.md (⭐️ recommended for project context)

A plain `.md` file at the repo root, loaded into the system prompt at session start. That's the whole architecture. 

```text
my-project/
├── CLAUDE.md                          # loaded by Claude Code
├── AGENTS.md                          # loaded by everything else
├── .cursorrules                       # used by Cursor
└── .github/copilot-instructions.md    # used by GitHub Copilot
```

Why it wins for project-scoped context — coding standards, build commands, domain language, conventions:

* **Deterministic** - the memory is always in context. No retrieval failure mode.
* **Version-controlled** - commit it to git, review it in a PR, diff it over time.
* **Human-editable** - no service to run, no embedding to recompute.
* **Free and cross-tool** - every agent harness honours it.

The ceiling is real (token cost grows linearly with the file, no semantic retrieval, no per-user state), but most teams hit the ceiling much later than they think. Newer setups layer `@import` directives so a top-level file pulls in scoped fragments.

### Claude Code's auto memory (⭐️ recommended baseline for Claude)

Auto memory is a **separate system** that lives alongside `CLAUDE.md`. Claude accumulates its own notes across sessions — build commands it discovered, debugging insights, code style preferences it picked up from your corrections — and stores them in a dedicated memory directory, not in your repo.

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # concise index — loaded into every session (first 200 lines / 25KB)
├── debugging.md       # detailed notes — loaded on demand when Claude needs them
├── api-conventions.md
└── ...
```

How it relates to CLAUDE.md:

|                      | CLAUDE.md files                           | Auto memory                                     |
| -------------------- | ----------------------------------------- | ----------------------------------------------- |
| **Who writes it**    | You                                       | Claude                                          |
| **What it contains** | Instructions and rules                    | Learnings and patterns it discovered            |
| **Scope**            | Project, user, or org                     | Per-repository, shared across worktrees         |
| **Loaded into**      | Every session (in full)                   | Every session (first 200 lines of `MEMORY.md`)  |
| **Use it for**       | Coding standards, workflows, architecture | Build commands, debugging insights, preferences |

Practical notes that matter:

* **On by default.** Toggle via `/memory` in a session, or `"autoMemoryEnabled": false` in settings, or `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`.
* **Machine-local.** Files are not shared across machines or cloud environments. All worktrees of the same repo share one memory directory.
* When you say `"remember X"` Claude writes to auto memory. When you say `"add this to CLAUDE.md"` it edits CLAUDE.md instead. Different words → different store.
* Run `/memory` any time to browse, audit, or edit what's been saved. Everything is plain markdown.

### Mem0

[Mem0](https://mem0.ai) sits on top of a vector store with an LLM-extraction layer. It pulls memorable facts from raw conversation, deduplicates against existing memories, and decides whether each new fact is an update or a new entry. Easiest to plug in. Best for **per-user personalisation** in chatbots and assistants — different problem from engineering conventions.

### Zep / Graphiti

[Zep](https://www.getzep.com) (and its open-source core, [Graphiti](https://github.com/getzep/graphiti)) is a **temporal knowledge graph**. Relationships have validity windows, so it can answer queries like "what was the policy as of last March" — which vector search just can't. Trade-off: slower writes, more setup, more complex mental model.

### Letta (formerly MemGPT)

[Letta](https://www.letta.com) uses an OS-style paging model — "core memory" always in context, "archival memory" sitting in a vector store the agent queries with `archival_memory_search` and `core_memory_replace`. The agent is in charge of its own memory. Best for **long-horizon autonomous agents** that run for days; the cost is committing to Letta as your harness instead of Claude Code.

### Beads (⭐️ recommended for engineering work)

[Beads](https://github.com/gastownhall/beads) (Steve Yegge, October 2025) is a distributed graph issue tracker for coding agents, backed by [Dolt](https://github.com/dolthub/dolt). The primary unit isn't a "memory," it's an **issue with typed dependencies** (`blocks`, `related`, `parent-child`, `supersedes`, `duplicates`). Free-form project knowledge is stored via `bd remember "insight"` and re-injected by `bd prime` at session start.

Best for **engineering work with real dependencies** and conventions worth pinning ("we resolve recipe alternates depth-first," "throughput rounds half-to-even at 6 decimals"). The hooks make `bd remember` fire automatically at decision points — roughly 60–80% reliable depending on model and session length — and `bd prime` makes retrieval rock-solid for everything that did get saved.

## How do you actually pick?

* **Coding agent on a specific repo** — start with `CLAUDE.md` (or `AGENTS.md` for cross-tool). Add nothing else until you feel actual pain.
* **Engineering work with conventions worth pinning** — layer Beads on top. The hooks make the save step decently reliable; `bd prime` makes retrieval bulletproof for what got saved.
* **Chatbot or assistant with many users** — Mem0. Drop-in SDK, handles per-user state.
* **Agents that need temporal reasoning** — Zep or Graphiti. "What was true last Tuesday" only works with validity windows.
* **Long-running autonomous agents (days, not minutes)** — Letta. The agent-managed paging model is genuinely different.
* **Already on LangChain / LangGraph** — LangMem. The integration tax for anything else is high.

## How close are we to a "human teammate" feel?

What people actually want when they ask for "agent memory" is a teammate: *I tell them once, from then on they remember it*. No system on the market delivers that end-to-end yet. The honest reframing is that the problem splits in two:

* **The save step** — does the agent capture what was decided?
* **The recall step** — does the right memory show up at the right moment?

Different systems fail at different halves:

|                    | Save step                                                      | Recall step                                      |
| ------------------ | -------------------------------------------------------------- | ------------------------------------------------ |
| Plain CLAUDE.md    | Manual ("save this") — 100% reliable if you remember to say it | Deterministic — always in context                |
| Claude auto-memory | Automatic-ish — quality varies                                 | Deterministic — always in context                |
| Beads (with hooks) | \~60–80% automatic via hooks                                   | Deterministic — `bd prime` injects it            |
| Mem0               | LLM-extracted, fuzzy                                           | Vector similarity — sometimes misses             |
| Letta              | Agent-managed, closest to automatic                            | Agent-prompted — works if it remembers to search |

<boxEmbed
  style="info"
  body={<>
    **Reframe that helps:** a human teammate forgets at the *recall* step. An agent forgets at the *save* step. The save step is easier to fix — you say it once, it's pinned, it stays pinned. The remaining 10% of "remember this" nudges is the actual cost of working with an AI teammate right now. It'll close over time, but it hasn't yet.
  </>}
  figurePrefix="none"
  figure=""
/>

The realistic stack today, ranked by effort:

1. **Start with `CLAUDE.md` / `AGENTS.md`. **Turn on auto-memory if you're using Claude Code. If it remembers enough of the right things, you're done.
2. **Layer Beads once auto-memory hits its limits.** Hooks for the save step, `bd prime` for retrieval. This is probably where you land for engineering work.
3. **Add Mem0 only if you also need per-user state.** Different problem from engineering conventions.
4. **Letta if you're willing to change agent runtimes.** Most "automatic" of the bunch but means leaving Claude Code.
