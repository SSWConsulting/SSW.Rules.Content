LLMs don't remember anything between turns. Every chat starts cold. What feels like an ongoing conversation is really one ever-growing context window getting re-sent to the model each time you hit enter. And the bigger that window gets, the more you pay for it, and the more the model starts to lose the plot.

"Agentic memory" boils down to one question: what should we store, where, and how do we pull it back when we need it?

## What kinds of memory are there?

The cognitive types come from the CoALA framework (Princeton, 2023):

* **Working / in-context** — the current conversation window. RAM. Close the session and it's gone.
* **Episodic** — what happened before. "The user asked about Resend in March."
* **Semantic** — generalised facts. "Gordon prefers self-hosted tooling."
* **Procedural** — how to do things. Workflows, learned routines, the agent's "muscle memory."

<boxEmbed
  body={<>
    Most production agents only really need two: working memory (handled by the LLM provider) plus some kind of long-term memory that blends the other three. Splitting them into separate stores is usually over-engineering until you have a concrete reason.

## How do you give your AI Agent memory?

There are dozens of options. Here are the ones that matter.

### CLAUDE.md / AGENTS.md

(\*\*Read more about… link rule)

A plain `.md` file at the repo root, loaded into the system prompt at session start. That's the whole architecture.

```text
my-project/
├── CLAUDE.md                          # loaded by Claude Code
├── AGENTS.md                          # loaded by everything else
├── .cursorrules                       # used by Cursor
└── .github/copilot-instructions.md    # used by GitHub Copilot
```

Why it works for project context (coding standards, build commands, domain language, conventions):

* Deterministic. The memory is always in context, so there's no retrieval failure mode.
* Version-controlled. Commit it to git, review it in a PR, diff it over time.
* Human-editable. No service to run, nothing to re-embed.
* Free and cross-tool. Every agent harness honours it.

There's a ceiling, of course. Token cost scales with the file, there's no semantic retrieval, no per-user state. But most teams hit that ceiling far later than they expect. Newer setups push it back further with `@import` directives, so a top-level file pulls in scoped fragments.

### Claude Code's auto memory (⭐️ recommended baseline for Claude)

Auto memory is a separate system that lives alongside `CLAUDE.md`. Claude takes its own notes across sessions (build commands it discovered, debugging insights, code style preferences it picked up from your corrections) and stores them in a dedicated memory directory, not in your repo.

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # concise index — loaded into every session (first 200 lines / 25KB)
├── debugging.md       # detailed notes — loaded on demand when Claude needs them
├── api-conventions.md
└── ...
```

How it relates to CLAUDE.md:

|                      | CLAUDE.md files                           | Auto memory                                     |
| -------------------- | ----------------------------------------- | ----------------------------------------------- |
| **Who writes it**    | You                                       | Claude                                          |
| **What it contains** | Instructions and rules                    | Learnings and patterns it discovered            |
| **Scope**            | Project, user, or org                     | Per-repository, shared across worktrees         |
| **Loaded into**      | Every session (in full)                   | Every session (first 200 lines of `MEMORY.md`)  |
| **Use it for**       | Coding standards, workflows, architecture | Build commands, debugging insights, preferences |

A few things worth knowing:

* It's on by default. You can toggle it with `/memory` in a session, set `"autoMemoryEnabled": false`, or use `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`.
* The files are machine-local. They don't sync across machines or cloud environments, though all worktrees of the same repo share one memory directory.
* Say `"remember X"` and Claude writes to auto memory. Say `"add this to CLAUDE.md"` and it edits CLAUDE.md instead. Different words, different store.
* Run `/memory` any time to browse or edit what's saved. It's all plain markdown.

### Mem0

[Mem0](https://mem0.ai) sits on top of a vector store with an LLM-extraction layer. It pulls facts from raw conversation, deduplicates against what's already there, and decides if each new fact is an update or a new entry. Easiest one to plug in. It's aimed at per-user personalisation in chatbots and assistants, which is a different problem from engineering conventions.

### Zep / Graphiti

[Zep](https://www.getzep.com) (and its open-source core, [Graphiti](https://github.com/getzep/graphiti)) is a temporal knowledge graph. Relationships have validity windows, so it can answer queries like "what was the policy as of last March," which vector search just can't. The trade-off is slower writes, more setup, and a more complex mental model.

### Letta (formerly MemGPT)

[Letta](https://www.letta.com) uses an OS-style paging model. "Core memory" sits in context, and "archival memory" lives in a vector store the agent queries with `archival_memory_search` and `core_memory_replace`. The agent manages its own memory. Best fit for long-horizon autonomous agents that run for days. The cost is committing to Letta as your harness instead of Claude Code.

### Beads (⭐️ recommended for engineering work)

[Beads](https://github.com/gastownhall/beads) (Steve Yegge, October 2025) is a distributed graph issue tracker for coding agents, backed by [Dolt](https://github.com/dolthub/dolt). The primary unit is an issue with typed dependencies (`blocks`, `related`, `parent-child`, `supersedes`, `duplicates`), rather than a free-form memory blob. Project knowledge is stored via `bd remember "insight"` and re-injected by `bd prime` at session start.

Good fit for engineering work with real dependencies and conventions worth pinning ("we resolve recipe alternates depth-first," "throughput rounds half-to-even at 6 decimals"). The hooks make `bd remember` fire automatically at decision points, which is roughly 60–80% reliable depending on the model and the session length. `bd prime` then makes retrieval rock-solid for everything that did get saved.

## How do you actually pick?

* Start with `CLAUDE.md` or `AGENTS.md`. Turn on auto-memory if you're using Claude Code. If it remembers enough of the right things, you're done.
* Layer Beads once auto-memory hits its limits. Hooks handle the save step, `bd prime` handles retrieval. This is probably where you land for engineering work.
* Add Mem0 only if you also need per-user state. That's a different problem from engineering conventions.
* Add Zep or Graphiti if your agents need temporal reasoning. "What was true last Tuesday" only works with validity windows.
* Try Letta if you're willing to change agent runtimes. It's the most "automatic" of the bunch, but it means leaving Claude Code.
* Already on LangChain or LangGraph? Use LangMem. The integration tax for anything else is high.

Reach for a real memory store when you've got a specific question the simple approach can't answer, usually "remember this user across sessions" or "what was true at time X." Not because the architecture diagram looked cool.
