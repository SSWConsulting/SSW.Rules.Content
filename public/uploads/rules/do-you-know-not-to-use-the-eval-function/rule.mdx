The JavaScript `eval()` function evaluates the content of a text string and executes it as JavaScript code. While it appears convenient, it is one of the most dangerous and inefficient constructs in JavaScript and should be avoided.

<endIntro />

Most uses of `eval` want part of a variable name to be dynamic. Once you realise that all global variables are held in the `window` object, you can reference them directly without `eval`.

```javascript
eval('somevar' + num);
```

```javascript
window['somevar' + num];
```

## Security Risks (XSS)

`eval()` is a major **security vulnerability**. Any string passed to `eval` is executed with the full privileges of the calling script. If user-controlled data reaches `eval`, an attacker can execute arbitrary JavaScript — a classic **Cross-Site Scripting (XSS)** attack vector.

**Never pass user input or untrusted data to `eval()`.**

## Performance

`eval` prevents JavaScript engines from optimising the evaluated code. Modern JIT compilers cannot inline or optimise code paths that call `eval`, resulting in significantly slower execution.

## Safer Alternatives

| Instead of... | Use... |
|---|---|
| `eval('{"key":"value"}')` | `JSON.parse('{"key":"value"}')` |
| `eval('window.' + varName)` | `window[varName]` |
| Dynamic function creation | Refactor to a lookup table or strategy pattern |

```javascript
// Bad – eval to parse JSON
const data = eval('(' + jsonString + ')');

// Good – use JSON.parse
const data = JSON.parse(jsonString);
```

## Content Security Policy (CSP)

A strong [Content Security Policy](https://developer.mozilla.org/en-US/docs/Web/HTTP/CSP) can help mitigate eval-based attacks. Set `script-src` without `'unsafe-eval'` to block `eval()` at the browser level:

```http
Content-Security-Policy: script-src 'self'
```

This prevents `eval()` from executing even if an attacker injects a malicious string.
