Agentic Workflows embed AI agents directly into your CI/CD pipelines, allowing them to autonomously reason about and execute multi-step tasks triggered by events like pull requests, issues, or schedules. Unlike traditional automation that requires rigid, deterministic scripts, agentic workflows use natural language instructions that an AI agent interprets at runtime - making them ideal for repetitive tasks that involve ambiguity and would benefit from human-like judgement. GitHub's Agentic Workflows is a leading implementation of this pattern.

## What are Agentic Workflows?

Agentic Workflows are a form of natural language programming that incorporates AI agents into your development lifecycle. Rather than writing step-by-step scripts, you describe **what** you want done and **why**, and an AI agent figures out **how** to do it at runtime. This bridges the gap between rigid automation (which breaks when context changes) and the flexible reasoning a human would apply to tasks like issue management, PR reviews, test generation, documentation, and reporting.

### GitHub Agentic Workflows

GitHub's Agentic Workflows is one of the first platform-native implementations of this pattern, built on top of GitHub Actions.

GitHub Agentic Workflows use markdown files with frontmatter to define triggers, permissions, safe outputs, tools, and natural language instructions. The markdown is then compiled into a GitHub Actions workflow via the `gh aw` CLI extension. For the full specification and setup guide, see the [gh-aw Documentation](https://github.github.com/gh-aw/).

## How is this different from regular AI-generated workflows?

You may be thinking “Why not just ask an AI agent to generate my CI/CD pipeline?” The key difference is **when** the AI reasoning happens.

With AI-generated automation, the agent writes the workflow steps once, you commit the file, and from then on it runs as a static, deterministic script - the same as any other workflow.

With agentic workflows, the AI agent **reasons at runtime** every time the workflow is triggered. This is where “productive ambiguity” helps. For example, a human teammate can understand “read the coverage report and write tests to improve the coverage.” This isn't a deterministic instruction set - it's something an AI agent can interpret and execute differently each time, adapting to the current state of the codebase.

### Security by design

Because an AI agent is making runtime decisions, security becomes critical. Any agentic workflow implementation should follow these principles:

* **Least privilege** - agents should have read-only access by default, with write operations explicitly granted
* **Sandboxed execution** - agents should run in isolated environments with limited filesystem and network access
* **Output constraints** - define exactly what an agent is allowed to produce (e.g. comments, labels, PRs) and set limits
* **Audit trails** - all agent actions should be logged and auditable

This is a fundamental difference from simply invoking an AI agent inside a normal CI/CD pipeline, where agents run with whatever permissions you give the job and have no built-in guardrails.

GitHub Agentic Workflows apply these principles through scoped permissions, containerized sandboxing, a network firewall (AWF), and tool allowlisting. See the [gh-aw security documentation](https://github.github.com/gh-aw/) for details on their security model.

## When should you use Agentic Workflows?

Agentic Workflows are best suited for repetitive tasks within your development workflow that involve some ambiguity and would benefit from AI reasoning, rather than requiring strict, deterministic outputs. They fit naturally into your existing CI/CD pipeline, triggered by the same events you already use.

Common use cases include:

* GitHub Issue triage - Automatically labelling, summarizing, and routing incoming issues
* Documentation updates - Keeping README's and API docs in sync as the code changes with modifications
* Test generation - Analyzing coverage gaps and creating PRs proposing new test cases
* Daily reporting - Generating status reports summarizing recent activity
* Code quality - Suggesting refactoring improvements and filing PRs with fixes

## Getting started with GitHub Agentic Workflows

To set up GitHub Agentic Workflows, follow the official setup guide at [gh-aw Documentation](https://github.github.com/gh-aw/). You can also explore [example workflows](https://github.com/githubnext/agentics) to see the pattern in action.

## Example

Here is the natural language instructions portion of a GitHub Agentic Workflow that automatically detects and closes duplicate issues. Notice how the instructions read like what you'd tell a human teammate, not a deterministic script:

```markdown
# Duplicate Issue Detector

You are an AI agent that detects duplicate issues in this repository.
When a new issue is opened, you analyze the existing issue backlog to
find potential duplicates. If a duplicate is found, you close the new
issue with a reference to the original and notify relevant users.

## Your Task

1. **Read the newly opened issue** — get the title, body, and any
   mentioned users from the triggering issue.
2. **Search the issue backlog** — search for existing open AND closed
   issues that may cover the same topic. Try multiple search queries
   to cast a wide net (at least 2-3 different queries).
3. **Evaluate similarity** — for each candidate, compare title
   similarity, description overlap, and labels. Look for **semantic**
   duplicates (same underlying problem described differently).
4. **If a duplicate is found** — close the new issue with a comment
   linking to the original, and notify relevant users.
5. **If NO duplicate is found** — take no action.
```

For the full workflow file (including frontmatter for triggers, permissions, and safe outputs), see the [example workflows repository](https://github.com/githubnext/agentics).
