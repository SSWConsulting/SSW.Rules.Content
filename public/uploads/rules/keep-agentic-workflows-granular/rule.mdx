GitHub Agentic Workflows let you instruct agents using natural language, embedded CLI commands, and MCP tools. Each workflow should perform **only one action. **For example, creating a pull request, creating an issue, or modifying a database record via a custom safe output. You don't get the artifacts of a create action until the workflow has finished running, and a single action keeps runs predictable, auditable, and safe. For any destructive or write operation, express that action via safe outputs so the workflow's one outcome is explicit and bounded.

<endIntro />

## What counts as one action?

One action is a single, well-defined outcome some examples include: 

* Creating an issue, 
* Creating a pull request
* Database updates via a custom safe output (e.g. a database record). 

You should avoid workflows that use free-form instructions, or CLI commands in the workflow body to perform multiple write operations. Instead, define the single action using safe outputs instead.

## Decompose into sub-workflows when you need multiple actions

If your workflow needs more than one action, for example, creating GitHub issues and then acting on them, decompose into **sub-workflows** that:

* **Call each other:** For example, workflow A creates issues; workflow B is triggered or invoked to process issues by label
* **Listen for events:** From other agents or workflows

That way each workflow still performs a single action, and you orchestrate the overall behaviour by how sub-workflows invoke or react to one another.

```
---
description: >
  This workflow logs issues to update outdated documentation on the site.

on:
  workflow_dispatch:
    inputs:
      label_name:
        description: "GitHub label slug that ties this pipeline together (e.g. 'update-react-docs')."
        required: true

permissions: read-all

safe-outputs:
  create-issue:
    labels: ["${{ inputs.label_name }}"]
    title-prefix: "Content Audit:"
    max: 30

tools:
  github:
    lockdown: false
    toolsets: [issues, repos, labels]

post-steps:
  - name: Trigger Agent B (PR Creator)
    if: success()
    env:
      INPUT_LABEL_NAME: ${{ inputs.label_name }}
    # A second agentic workflow is triggered to action the recently created issues.
    run: |
      gh workflow run action-outdated-content-issues.lock.yml \
        -f label_name="$INPUT_LABEL_NAME" \
---

I want you to to create an issue in this repository after finding outdated content in the documentation.

Use the `create_issue` safe-output tool to create the issue.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full body in markdown

```

Figure: ✅ Good Example - This flow handles the responsibility of creating GitHub issues, but delegates the work to another agentic workflow.

## Use safe outputs for that single action

Safe outputs are how you model the desired behaviour of your agentic workflow. They provide safety against prompt injection and secure your workflow using the least privilege principle.

* **Controllable** - They provide a clear specification for what your workflow will produce, preventing unintentional side effects
* **Auditable** - Clearly visible in workflow logs and job summaries
* **Secure** - Using safe outputs for tool calls avoids giving the agent access to potentially dangerous secrets in your repository

MCP scripts are useful for **read** operations, such as using a search engine to gather information before performing an action. They are not recommended for write operations in GitHub Agentic Workflows because they bypass the audit trail that runs implicitly when your agent runs. If your workflow's single action is not covered by the default tool sets, define it using [Custom Safe Outputs](https://github.github.com/gh-aw/reference/custom-safe-outputs/).

````yaml
---
description: >
  Create an example an issue for adding telemetry to the site.

on:
  workflow_dispatch:

engine:
  id: copilot
  model: gpt-5-mini

permissions: read-all

---
I want you to create an issue in this repository for adding telemetry to the site. 

Use the command below to create an issue via the GitHub REST API:

```bash
curl -s -X POST \
  -H "Authorization: Bearer $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  "https://api.github.com/repos/${{ github.repository }}/issues" \
  -d '{
    "title": "<TITLE>",
    "body": "<BODY>"
  }'
```

Where the agent fills in:
- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown (JSON-escaped)

````

**Figure: ❌ Bad example - Using inline curl to create issues instead of a single, bounded safe output**

```yaml
---
description: >
  Create an example issue for adding telemetry to the site using
  safe-outputs instead of curl.

on:
  workflow_dispatch:


permissions: read-all

safe-outputs:
  create-issue:
    title-prefix: "[Example] "
    max: 1

tools:
  github:
    lockdown: false
    toolsets: [issues]
---
I want you to create an issue in this repository for adding telemetry to the site.

Use the `create_issue` safe-output tool to create the issue.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown
```

**Figure: ✅ Good example - The intended action of this workflow is defined within the safe outputs**
