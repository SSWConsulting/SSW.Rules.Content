A toast notification is small. Its impact is not.

Poor placement creates confusion. Good placement feels invisible and intentional.

If users hesitate for even a second wondering “Was that my system or the app?”, you have introduced friction.

For most modern web applications, there is a clear default: **Bottom right** is the safest and most predictable choice.

<endIntro />

## Analysis of all six positions

## 2. Analysis of All Six Positions

| Position | ✅ Pros | ❌ Cons |
|----------|------|------|
| Top Left | Highly visible; matches Western reading start point | Interrupts primary content scan (F-pattern); conflicts with navigation and logo area |
| Top Center | Very high visibility; matches iOS notification position; ideal for mobile-first apps | Blocks main heading and hero content; too disruptive for non-critical messages |
| Top Right | Visible without blocking primary content; common in older Bootstrap examples; familiar from Windows 7–10 era | Conflicts with macOS Notification Center; largely superseded convention; interrupts natural reading flow |
| Bottom Left | No OS conflict on any platform; familiar from Google products; does not interrupt reading | Opposite corner to most primary CTAs; slightly less prominent than bottom right in RTL layouts |
| Bottom Center | Material Design 3 standard; balanced and symmetrical; works well on mobile and desktop; no OS conflict | Can overlap bottom navigation bars or floating action buttons on mobile |
| **Bottom Right ⭐️** | Matches Windows convention; enterprise SaaS standard; avoids macOS top-right conflict; peripheral but noticeable | Minor conflict with Windows taskbar notification area (OS notifications appear as overlays above apps) |

## What the major design systems recommend

| Design System                                                                                                                       | Recommended Position   |
| ----------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| **[Google — Material Design 3](https://m3.material.io/components/snackbar/overview)**                                               | Bottom center          |
| **[shadcn/ui + Sonner](https://ui.shadcn.com/docs/components/sonner)**                                                              | Bottom right (default) |
| **[Bootstrap 5](https://getbootstrap.com/docs/5.3/components/toasts/)**                                                             | Top right (examples)   |
| **[Apple HIG (iOS)](https://developer.apple.com/design/human-interface-guidelines/notifications)**                                  | Top center             |
| **[Microsoft Windows](https://learn.microsoft.com/windows/apps/design/shell/tiles-and-notifications/toast-notifications-overview)** | Bottom right           |
| **[Apple macOS](https://developer.apple.com/design/human-interface-guidelines/notifications)**                                      | Top right              |

## What real web apps actually do

| App | Toast Position | Example |
|-----|---------------|----------|
| **Notion** | **Bottom right** | “Link copied” |
| **Slack** | **Bottom right** | Connection status |
| **Stripe** | **Bottom right** | Save confirmations |
| GitHub | Bottom center | “Copied!” |
| Figma | Bottom center | “Link copied” |
| X | Bottom center | “Tweet sent” |
| Gmail | Bottom left | “Message sent. Undo” |
| YouTube | Bottom left | “Added to playlist” |
| LinkedIn | Bottom left | “Invitation sent” |
| Spotify (web) | Bottom left | “Added to queue” |

### Pattern

* **Consumer apps** - Bottom left or bottom center
* **Enterprise and SaaS tools** - Bottom right

If you are building internal tools, admin portals, or SaaS dashboards, bottom right aligns with user expectation.

***

## Why top right is risky

Bootstrap examples often show top-right toasts.

However:

* macOS system notifications appear in the top right
* In-app toasts in that location compete visually with the OS
* Users may confuse app feedback with system alerts

Do **not** fight the operating system.

***

## Accessibility matters

Toast placement is not just visual. It is semantic.

Follow WCAG guidance:

* Use `role="status"` for success and informational toasts
  * Polite ARIA live region
  * Screen readers announce after the current task
* Use `role="alert"` for errors
  * Assertive
  * Screen readers interrupt immediately
* Minimum display time: 4 seconds (WCAG 2.2 guideline 2.2.1)
* Never auto-dismiss error toasts
* Do not cover primary CTAs or form fields

***

## Recommendation

If your web app targets both Windows and Mac users:

### ✅ Choose bottom right

* Matches Windows convention
* Avoids macOS notification area
* Used by Notion, Slack, and Stripe
* Default in shadcn/ui + Sonner

### ✅ Bottom center is a strong alternative

* Aligns with Material Design
* Works well for consumer apps
* Less likely to overlap navigation

### ❌ Avoid top right

Unless you are building a Windows-only enterprise system where bottom-right conflicts with taskbar notifications.

***

If you are building a modern SaaS or enterprise web application, default to **bottom right**.

Small details build trust. Predictability reduces friction.
