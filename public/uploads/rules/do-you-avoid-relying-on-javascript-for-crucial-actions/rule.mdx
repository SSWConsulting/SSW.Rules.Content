Don't assume JavaScript is always enabled.

JavaScript should be used to **progressively enhance** the user experience — not as a hard dependency for core functionality.

<endIntro />

JavaScript is very useful for improving user-interaction and reducing server requests, but it can be disabled or fail to load. Historically an estimated 2% of web users did not have JavaScript enabled; today, more relevant concerns include slow networks, corporate proxies, browser extensions, and partial failures in Single Page Applications (SPAs).

Depending on your audience you may choose to adapt this rule, but for mainstream websites it is highly recommended that you don't rely on JavaScript for crucial actions, such as **validation** or **business-logic** purposes. Always do server-side validation as the authoritative check.

## Progressive Enhancement

Build a functional baseline using HTML and server-side logic, then layer JavaScript on top to enhance the experience. This means:

- Forms should submit and validate without JavaScript (server-side validation is mandatory)
- Navigation should work without client-side routing
- Critical content should be rendered server-side (SSR) or statically, not only via client-side JS

## Single Page Applications (SPAs)

Modern SPAs (React, Angular, Vue) rely heavily on JavaScript. To handle this:

- Use **Server-Side Rendering (SSR)** or **Static Site Generation (SSG)** for initial page loads (e.g. Next.js, Nuxt, Angular Universal)
- Implement **API-first validation** — always validate on the server even when the client also validates
- Add a `<noscript>` fallback to inform users when JavaScript is required

```html
<noscript>
  <p>This application requires JavaScript. Please enable it to continue.</p>
</noscript>
```

## Accessibility Considerations

When JavaScript controls interactive elements (modals, accordions, dynamic content), ensure:

- **Focus management**: Move focus to the opened modal/dialog; return it when closed
- **ARIA attributes**: Use `aria-expanded`, `aria-haspopup`, `aria-modal`, and `role` attributes appropriately
- **Keyboard support**: All interactions must be operable via keyboard (Tab, Enter, Escape)

See [WAI-ARIA Authoring Practices](https://www.w3.org/WAI/ARIA/apg/) for detailed patterns.
