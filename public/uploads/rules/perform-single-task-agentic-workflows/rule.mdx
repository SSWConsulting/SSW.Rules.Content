GitHub agentic workflows, allow you to instruct agents tempting to perform complex instructions using a variety of tooling, including natural language instructions, embedded CLI commands, and MCP tools. However there's a major caveat to performing work with Agentic Workflows. You don't get the artifacts of a create action until the workflow has finished running.

In particular, when your tooling performs destructive actions, such as creating issues, or even using MCP to perform create, update, or delete actions elsewhere, you should restrict these actions to your Safe Outputs specification.

<endIntro />

## What are Safe Outputs?

Safe outputs are a how you model the desired behaviour of your agentic workflow. They provide safety against prompt injection and secure your workflow using the least privilege principle.

* **Controllable** - They provide a clear specification for what your workflow will produce, preventing unintentional side effects
* **Auditable** - Clearly visible in workflow logs and job summaries
* **Secure** - Using safe outputs for tool calls avoids giving the agent access to potentially dangerous secrets in your repository

## What if my Workflow performs multiple actions?

If your workflow needs to perform multiple actions, such as creating GitHub issues and then actioning them, you should chain your Agentic workflows together. For example, after your agent finishes creating GitHub issues you could have a sub-agent that searches the new issues by label to perform the work.

## Why not use MCP tools?

MCP scripts are very useful for allowing your workflow to perform read operations reliably. For example your workflow may need to use a search engine to collect more information before performing an action. However they are not recommended for write operations in GitHub Agentic Workflows. The reason for this is they bypass the audit trail that runs implicitly whenever your agent runs.

If your workflow run needs to result in an action that is not specified by the default tool sets you should define the tooling for performing that action using [Custom Safe Outputs](https://github.github.com/gh-aw/reference/custom-safe-outputs/).

````yaml
---
description: >
  Create an example an issue for adding telemetry to the site.

on:
  workflow_dispatch:

engine:
  id: copilot
  model: gpt-5-mini

permissions: read-all

---
I want you to create an issue in this repository for adding telemetry to the site. 

Use the command below to create an issue via the GitHub REST API:

```bash
curl -s -X POST \
  -H "Authorization: Bearer $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  "https://api.github.com/repos/${{ github.repository }}/issues" \
  -d '{
    "title": "<TITLE>",
    "body": "<BODY>"
  }'
```

Where the agent fills in:
- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown (JSON-escaped)

````

**Figure: ❌ Bad example - Creating GitHub issues using curl in the body of a flow**

```yaml
---
description: >
  Create an example issue for adding telemetry to the site using
  safe-outputs instead of curl.

on:
  workflow_dispatch:


permissions: read-all

safe-outputs:
  create-issue:
    title-prefix: "[Example] "
    max: 1

tools:
  github:
    lockdown: false
    toolsets: [issues]
---
I want you to create an issue in this repository for adding telemetry to the site.

Use the `create_issue` safe-output tool to create the issue.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown
```

**Figure: ✅ Good example - Defining Issue creation as a safe output and using the built in toolset**
