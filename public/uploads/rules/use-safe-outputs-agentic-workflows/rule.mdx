GitHub agentic workflows, allow you to instruct agents tempting to perform complex instructions using a variety of tooling, including natural language instructions, embedded CLI commands, and MCP tools. However there are some caveats as to when and where your agent should run code to solve problems.

In particular, when your tooling performs destructive actions, such as creating issues, or even using MCP to perform create, update, or delete actions elsewhere, you should restrict these actions to your Safe Outputs specification.

<endIntro />

## What are Safe Outputs?

Safe outputs are a how you model the desired behaviour of your agentic workflow. They provide safety against prompt injection and secure your workflow using the least privelege principle.

* Controllable: They provide a clear specification for what your workflow will produce, preventing unintentional side effects
* **Auditable:** Clearly visible in workflow logs and job summaries
* **Secure:** Using safe outputs for tool calls avoids giving the agent access to potentially dangerous secrets in your repository.

## Why not use MCP scripts?

MCP scripts are very useful for reliable read operations on your repo, for example fetching data from an external API that requires authentication. repeatable instructions to your agent, but they are not recommended for write operations in GitHub Agentic Workflows. ## Examples

````yaml
---
description: >
  Create an example an issue for adding telemetry to the site.

on:
  workflow_dispatch:

engine:
  id: copilot
  model: gpt-5-mini

permissions: read-all

---
I want you to create an issue in this repository for adding telemetry to the site. 

Use the command below to create an issue via the GitHub REST API:

```bash
curl -s -X POST \
  -H "Authorization: Bearer $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  "https://api.github.com/repos/${{ github.repository }}/issues" \
  -d '{
    "title": "<TITLE>",
    "body": "<BODY>"
  }'
```

Where the agent fills in:
- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown (JSON-escaped)

````

**Figure: ❌ Bad example - creating GitHub issues using curl in the body of a flow**

```yaml
---
description: >
  Create an example issue for adding telemetry to the site using
  safe-outputs instead of curl.

on:
  workflow_dispatch:


permissions: read-all

safe-outputs:
  create-issue:
    title-prefix: "[Example] "
    max: 1

tools:
  github:
    lockdown: false
    toolsets: [issues]
---
I want you to create an issue in this repository for adding telemetry to the site.

Use the `create_issue` safe-output tool to create the issue.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown
```

**Figure: ✅ Good example - Defining Issue creation as a safe output and using the built in toolset**

## Key principles

1. **Emit structured data, not prose:** use `$GITHUB_OUTPUT`, JSON files, or job summaries
2. **Avoid MCP calls for workflow control:** MCP is great for giving agents *context*, not for *driving* workflow steps
3. **Reference outputs explicitly:** use `steps.<id>.outputs.<key>` so the relationship is visible in the workflow definition
4. **Keep agent logic separate from workflow logic:** the agent decides, the workflow acts
