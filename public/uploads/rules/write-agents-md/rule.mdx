AI coding assistants like Cursor and GitHub Copilot are incredibly powerful, but they work best when they understand your project's specific architecture, coding standards, and common tasks. Without this context, they may suggest patterns that don't align with your codebase.

`AGENTS.md`is an agreed-upon standard between all the big AI players, replacing things like `.cursorrules` and `copilot_instructions.md`.

## What goes into an AGENTS.md?

An `AGENTS.md` file acts as a "manual" for AI agents. It provides a centralized place to define:

* **âœ… Architecture overview** - Explain the tech stack and how different layers interact
* **âœ… Coding standards** - Define naming conventions, preferred libraries, and patterns
* **âœ… Common commands** - List frequently used scripts for building, testing, and linting
* **âœ… Key patterns** - Provide code examples for common tasks (e.g. adding a new API endpoint)
* **âœ… Links to documentation** - Reference ADRs, design docs, or wikis rather than duplicating content (e.g. `See /docs/adr/001-authentication.md for auth patterns`)
* âŒ **Secrets** - Never put API keys, secrets, or sensitive internal information, it's committed to source control

By providing this information, you ensure that the AI's suggestions are consistent with your project's existing structure, reducing the need for manual corrections.

## Even better: multiple AGENTS.md

You might be thinking: â€œWait.. I have a massive monorepo. There's no way it's efficient to capture all that information in one file, and feed it into the AI in one goâ€¦â€

Correct. It's unwise to waste precious space in the context window, but dumping in the entire AGENTS.md if only 5% of it is useful.

This is where hierarchy comes in. As per the standard, Agents will prioritise the nearest `AGENTS.md` file, while also injecting all parent `AGENTS.md` in its path.

<imageEmbed alt="Image" size="large" showBorder={false} figurePrefix="none" figure="If working in `repo/apps/web-app/src`, the highlighted AGENTS.md are used" src="/uploads/rules/write-agents-md/filepathagentsmd.png" />

In the case of conflicting or more specific instructions, the closest `AGENTS.md` file in whichever directory the Agent works in will take precedence. This means every subfolder can have its own instructions, without sacrificing detail.

The "root" `AGENTS.md` file in this case should contain only high-level instructions for what each package is for, what the overall architecture is, and how to run the whole application, as the detailed instructions will be in the package-specific `AGENTS.md` files.

Multiple AGENTS.md is particularly useful for architectures like [Modular Monoliths](/modular-monolith-architecture), where you can place AGENTS.md inside each module.

## Writing tips

* **Be concise** - Put the most important rules first; long files may be truncated
* **Use imperative language** - "Use X" not "You should consider using X"
* **Show, don't tell** - Specific code examples beat vague descriptions
* **Avoid duplication** - Don't repeat what's in linter configs or `.editorconfig`
* **Keep it current** - Outdated instructions are worse than none
