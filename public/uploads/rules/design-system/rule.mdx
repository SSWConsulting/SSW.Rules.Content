When every screen looks a little different and handoffs are painful, you know something's wrong.

**A design system bridges the gap **between design and engineering, aligning everyone around a shared language and providing the tools to build faster and better. It avoids inconsistent UI, longer review cycles, and time spent debating minor visual tweaks.

And now that AI agents are writing UI too, it decides whether an agent builds from your real components or invents new ones every time.

## The UI pain - signs you need a design system

You usually feel the absence of a design system before you can name it. A few signs:

* UI styling (e.g spacing, font, colour, radius) shifts between screens for no reason
* A "small" change in one place turns into a hunt through the codebase
* Design intent doesn't match what's actually in production 
* Handoffs and review meetings are dominated by nitpicking visual issues
* Style guides are outdated, scattered, or missing entirely

## What is a design system?

A design system is the shared source your team designs and builds from. Usually containing the following:

* **Tokens** - Small, reusable decisions underneath everything: colour, spacing, fonts, corner radius. Define them once and every component inherits them.
* **Component library** - Buttons, inputs, modals, and patterns themselves, with the variants and the rules for using them. The version that actually pays off lives in code, not only in a design file.
* **Documentation** - Guidelines for component usage, dos and don’ts, examples and demos
* **Governance** - Clear ownership of the design system, a contribution process, tools for dev handover, and semantic versioning to manage updates

## Levels of design systems

People use "design system" to mean anything from a single design file to a full coded platform, so it helps to know which one you actually have:

| Stage                                     | Description                                                                                                                                     |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **1. Design Library**                     | A design file (e.g. Figma) with consistent styles and reusable components like buttons and inputs.                                              |
| **2. Documented System (e.g. SSW Rules)** | Adds usage guidelines, naming conventions, dos and don'ts, accessibility considerations, and basic governance (who owns updates).               |
| **3. Design–Dev Alignment**               | Design tokens are mapped to code, creating a shared language between design and engineering for colour, spacing, and other foundational styles. |
| **4. Coded Components**                   | Components exist in code (e.g. Storybook) and match the Design Library.                                                                         |
| **5. Scalable Ecosystem**                 | Formal governance, versioning, a contribution model, changelogs, and multi-product usage. Often has its own site or portal.                     |

## Where AI raises the stakes - and where coded components matter most

Agentic workflows empower developers to cover more ground, producing a great volume of work at record speed. So without a system developers can actually build from, they will inevitably choose to build the UI from scratch, now that they're enabled to with the power of AI. It's now easier than ever to make a mess of things 10x faster. A coded design system changes what the agent has to work with. Instead of inventing components, it assembles the ones you have already designed, reviewed, and tested. We build ours in Storybook, which an agent can read through MCP (the protocol that lets it pull your real components into context). The agent still moves quickly. It just stays inside the boundaries you have already set. Building in code gives you a safety net for the same reason. Visual diffs run on every pull request, so a change that quietly shifts an unrelated component gets flagged before it merges, and you can run an accessibility audit across the whole library at once.

## How to set up a design system 

You do not need the whole thing on day one. Most teams start with a handful of shared components and grow from there. A sensible order:

Remember, you can start small - eventually you will cross the threshold from documentation into infrastructure. 

Start small enough to ship something useful quickly, then expand. A shared library with a few solid components already saves time. The coded layer is where it starts paying off, for your developers and your AI agents alike.

1. **Audit what you have** - Look for visual and structural inconsistencies
2. **Choose a direction **- Build a bespoke design system from the ground up - or use an open-source one and tweak it to match your brand.
3. **Define your tokens **- Start with the colour, spacing, and type values unique to your brand. 
4. **Build your first components** - Focus on core UI elements (i.e. header, footer, buttons, input fields)
5. **Get the tokens into code** - Tools like [Style Dictionary](https://styledictionary.com/) can help push your tokens into code.
6. **Move the components into code too** - Somewhere both developers and agents can pull from, like [Storybook](https://storybook.js.org/).
7. **Document as you go** - Usage notes, examples, and versioning.
8. **Set up ownership and checks** - Decide who maintains it, then add visual regression and accessibility audits so it stays honest.
