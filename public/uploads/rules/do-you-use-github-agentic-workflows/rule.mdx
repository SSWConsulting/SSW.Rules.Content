GitHub Agentic Workflows let you automate repository tasks by writing natural language instructions in markdown. These markdown files then compile into GitHub Actions workflows powered by AI agents. Instead of writing complex YAML, you describe what you want done and an AI agent handles it.

<endIntro />

## What are GitHub Agentic Workflows?

GitHub Agentic Workflows are a new automation layer built on top of GitHub Actions. You write a markdown file with YAML frontmatter describing **triggers**, **permissions**, **safe outputs**, and **tools**, followed by natural language instructions. The `gh aw` CLI extension then compiles this into a lock file, which is similar to a GitHub Actions YAML file.

The key difference from traditional CI/CD is that instead of scripting every step, you describe the desired behavior in plain English and an AI agent (such as GitHub Copilot, Claude Code, or OpenAI Codex) interprets and executes it.

A workflow file has four key parts:

* **Triggers** (`on:`) - Standard GitHub Actions events (push, schedule, issue opened, etc.)
* **Permissions** (`permissions:`) - Scoped access controls, read-only by default
* **Safe outputs** (`safe-outputs:`) - A way for the agent to delegate potentially dangerous tasks safely (e.g. creating a PR or issue, writing a comment, etc.)
* **Tools** (`tools:`) - Operations the agent can use (e.g. edit, web-fetch)

The instructions section is plain markdown where you describe the agent's role, goals, and steps.

## How is this different from AI-generated workflows?

You might be thinking - "Why not just ask my AI agent to write me a GitHub Actions YAML file?" The key difference is **when** the AI is involved.

With AI-generated YAML, the AI does its work at **authoring time**. It writes the workflow steps, you commit them, and from that point on, it runs like any other static workflow - the same steps execute in the same order, every time. It's a one-shot generation.

With agentic workflows, the AI reasons at **runtime**. It reads your instructions, evaluates the current state of the repository, and dynamically decides what actions to take. This means it can adapt - triaging different issues differently, writing contextual comments based on what it reads, or skipping steps that aren't relevant.

This makes agentic workflows powerful for tasks that require judgment and adaptation rather than rigid step-by-step execution.

### Security by design

Because an AI agent is making runtime decisions, security becomes critical. GitHub Agentic Workflows enforce a defense-in-depth approach:

* **Read-only by default** - The agent has no write access unless explicitly granted through `safe-outputs`
* **Permission separation** - Write operations run in separate permission-controlled jobs, never inside the agentic container itself
* **Sandboxed execution** - The agent runs in a containerized sandbox within the GitHub Actions VM, with limited filesystem access
* **Network isolation** - Egress traffic is restricted to a configurable domain allowlist via the Agent Workflow Firewall (AWF)
* **Tool allowlisting** - Agents can only use pre-approved tools, preventing unauthorized actions

This is a fundamental difference from AI-generated YAML, where the resulting workflow runs with whatever permissions you give the job and has no built-in guardrails on what the AI suggested.

## When should you use GitHub Agentic Workflows?

Agentic Workflows are best suited for repetitive repository maintenance tasks that benefit from AI reasoning, and do not require strict, deterministic outputs. They particularly shine for open-source maintainers dealing with high-volume repositories.

Common use cases include:

* **Issue triage** - Automatically label, summarize, and route incoming issues
* **Documentation updates** - Keep READMEs and API docs in sync as code evolves
* **Test generation** - Analyze coverage gaps and propose new test cases via PRs
* **Daily reporting** - Generate status reports summarizing recent activity
* **Code quality** - Suggest refactoring improvements and file PRs with fixes

Here's an example workflow that automatically detects and closes duplicate issues:

```markdown
---
description: >
  Automatically detect duplicate issues and close them with a link
  to the original, notifying relevant users.
on:
  issues:
    types: [opened]
permissions:
  contents: read
  issues: read
  pull-requests: read
tools:
  github:
    toolsets: [default]
safe-outputs:
  add-comment:
    max: 3
  close-issue:
    max: 1
  update-issue:
    max: 2
  noop:
    max: 1
---

# Duplicate Issue Detector

You are an AI agent that detects duplicate issues in this repository.
When a new issue is opened, you analyze the existing issue backlog to
find potential duplicates. If a duplicate is found, you close the new
issue with a reference to the original and notify relevant users.

## Your Task

1. **Read the newly opened issue** — get the title, body, and any
   mentioned users from the triggering issue.

2. **Search the issue backlog** — use the GitHub tools to search for
   existing open AND closed issues that may cover the same topic.
   Try multiple search queries to cast a wide net (at least 2-3
   different queries).

3. **Evaluate similarity** — for each candidate, compare title
   similarity, description overlap, and labels. Look for **semantic**
   duplicates (same underlying problem described differently).

4. **If a duplicate is found**:
   - Comment on the NEW issue explaining it's a duplicate, linking
     to the original with `#<number>`
   - Close the NEW issue as "not planned"
   - Comment on the ORIGINAL issue tagging the reporter and any
     mentioned users for visibility
   - Add the `duplicate` label to the new issue

5. **If NO duplicate is found**:
   - Call the `noop` safe output with a summary message.

## Guidelines

- **Be conservative**: Only close if you are confident it is truly
  a duplicate. When in doubt, leave it open.
- **Prefer the older issue**: Always close the newer one.
- **Be helpful**: Comments should be polite and explain why you
  believe it's a duplicate.
```

## How do you set up GitHub Agentic Workflows?

### Option 1: Agentic - Vibe it up! 🤙

1. Prompt your AI Agent to write you an Agentic Workflow with the following prompt:

```markdown
Create a workflow for GitHub Agentic Workflows using https://raw.githubusercontent.com/github/gh-aw/main/create.md

{{ YOUR WORKFLOW SPECIFICATION HERE }}
```

### Option 2: Handcraft your prompt 🤓

If you prefer to manually create your Agentic Workflows, follow these instructions. Learn more at [GitHub Agentic Workflow Docs - Manual Editing](https://github.github.com/gh-aw/setup/creating-workflows/#manual-editing)

1. Install the `gh aw` GitHub CLI extension
2. Write a `.md` file with YAML frontmatter (triggers, permissions, safe-outputs, tools) followed by natural language instructions
3. Run `gh aw` to generate the GitHub Actions `.lock.yml` file from your markdown
4. Add both the `.md` source and the generated `.lock.yml` to your repository
5. The workflow runs on the events you specified, with the AI agent executing your instructions
