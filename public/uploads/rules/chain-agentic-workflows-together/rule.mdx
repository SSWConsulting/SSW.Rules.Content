GitHub Agentic Workflows let you instruct agents using natural language, embedded CLI commands, and MCP tools. You don't get the artefacts of a create action until the workflow has **finished** running. So when you have steps that **depend on one another**—for example, Step 2 needs the issues or PRs that Step 1 creates—those steps cannot run in the same workflow. Create **new workflows** for dependent steps: let the first workflow complete and produce its artifacts, then trigger or invoke a second workflow that can use them. For any destructive or write operation, express it via safe outputs so each workflow's outcome is explicit and bounded.

<endIntro />

## When do steps depend on one another?

Steps depend on one another when a later step needs the **output or artefacts** of an earlier step. For example, you might create a number of GitHub issues, then have a separate step in your flow that performs the work outlined in those isses.

Because artefacts are only available after a workflow run completes, dependent steps must run in **separate workflows**. Keep steps that depend on one another out of the same workflow body.

## Create new workflows for dependent steps

When you have steps that depend on one another, put them in **separate workflows** that:

* **Call each other:** For example, workflow A creates issues and in a post step runs workflow B; workflow B is triggered with a label name and processes the newly created issues
* **Listen for events:** One workflow emits or updates something that another workflow is triggered on

Chain workflows so that the workflow that produces the artefacts runs first, then the workflow that consumes them runs after.

```
---
description: >
  This workflow logs issues to update outdated documentation on the site then performs the work to fix them.

on:
  workflow_dispatch:
    inputs:
      label_name:
        description: "GitHub label slug that ties this pipeline together (e.g. 'update-react-docs')."
        required: true

safe-outputs:
  create-pull-request:
    title-prefix: "[Content Fixer] "
    max: 5
  create-issue:
    title-prefix: "[Content Audit] "
    max: 5

permissions: read-all

tools:
  github:
    lockdown: false
    toolsets: [issues, repos, labels, pull_requests]

---

### Step 1

I want you to to create an issues in this repository after finding outdated content in the documentation.

Use the create issue safe-output tool to create the issues.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full body in markdown

### Step 2

After you've finished creating the issues in step 1 I want you to create pull requests using the pull requests safe-output tool to implement the changes.
Link to each issue by ID in each pull request respectively.

```

```
---
description: >
  This workflow logs issues to update outdated documentation on the site.

on:
  workflow_dispatch:
    inputs:
      label_name:
        description: "GitHub label slug that ties this pipeline together (e.g. 'update-react-docs')."
        required: true

permissions: read-all

safe-outputs:
  create-issue:
    labels: ["${{ inputs.label_name }}"]
    title-prefix: "Content Audit:"
    max: 30

tools:
  github:
    lockdown: false
    toolsets: [issues, repos, labels]

post-steps:
  - name: Trigger Agent B (PR Creator)
    if: success()
    env:
      INPUT_LABEL_NAME: ${{ inputs.label_name }}
    # A second agentic workflow is triggered to action the recently created issues.
    run: |
      gh workflow run action-outdated-content-issues.lock.yml \
        -f label_name="$INPUT_LABEL_NAME" \
---

I want you to to create issues in this repository after finding outdated content in the documentation.

Use the create issues safe-output tool to create the issue.

- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full body in markdown

```

Figure: ✅ Good example - Dependent steps split into separate workflows: this workflow creates issues, then triggers another workflow that can use those issues

## Use safe outputs for each workflow's actions

Safe outputs are how you model the desired behaviour of each agentic workflow. They provide safety against prompt injection and secure your workflow using the least privilege principle.

* **Controllable** - They provide a clear specification for what your workflow will produce, preventing unintentional side effects
* **Auditable** - Clearly visible in workflow logs and job summaries
* **Secure** - Using safe outputs for tool calls avoids giving the agent access to potentially dangerous secrets in your repository

MCP scripts are useful for **read** operations, such as using a search engine to gather information before performing an action. They are not recommended for write operations in GitHub Agentic Workflows because they bypass the audit trail that runs implicitly when your agent runs. If a workflow's action is not covered by the default tool sets, define it using [Custom Safe Outputs](https://github.github.com/gh-aw/reference/custom-safe-outputs/).

````yaml
---
description: >
  Create an example an issue for adding telemetry to the site.

on:
  workflow_dispatch:

engine:
  id: copilot
  model: gpt-5-mini

permissions: read-all

---
I want you to create an issue in this repository for adding telemetry to the site. 

Use the command below to create an issue via the GitHub REST API:

```bash
curl -s -X POST \
  -H "Authorization: Bearer $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  "https://api.github.com/repos/${{ github.repository }}/issues" \
  -d '{
    "title": "<TITLE>",
    "body": "<BODY>"
  }'
```

Where the agent fills in:
- `<TITLE>` — a concise issue title describing the request
- `<BODY>` — the full issue body in markdown (JSON-escaped)

````
